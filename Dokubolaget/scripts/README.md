# Board Scripts Workflow

> This README was generated by GitHub Copilot (GPT-5.3-Codex).

This folder contains the 3-step board pipeline for Dokubolaget.

## Prerequisites

- Run commands from `Dokubolaget/`
- `products.json` must exist in the repository root (one level above `Dokubolaget/`)

## Dev API/Proxy Notes

- The board scripts read from the local `products.json` snapshot and do not call the API directly.
- To refresh or debug API data in app/dev flows, run the local proxy:

```bash
bun run proxy
```

- Keep using the project proxy setup for Systembolaget API calls in development.
- If you regenerate `products.json` from API data, rerun all 3 board steps afterward.

## Step 1: Find Tags

Build a semi-restricted list of viable tags that can pair with many other tags.

```bash
bun run find:tags --min-cell 4
```

Output file:

- `data/board-tags.json`

What `--min-cell` means:

- Minimum products required for a tag-pair intersection to count as playable.
- Higher value = stricter tags, fewer but safer boards.

## Step 2: Generate Seeded Boards

Generate one or more 3x3 boards from viable tags.

```bash
bun run generate:board --seed 2026-04-12 --min-cell 4 --attempts 5000 --boards 3
```

Output file:

- `data/generated-boards.json`

Useful flags:

- `--seed`: deterministic daily board key
- `--min-cell`: minimum products per cell
- `--attempts`: search effort for better boards
- `--boards`: number of board options to output

## Step 3: Confirm Board Counts

Recompute and print exact solution counts for each of the 9 cells.

```bash
bun run confirm:board --board-index 0
```

Optional custom board file:

```bash
bun run confirm:board -- --board-file data/generated-boards.json --board-index 0
```

## Recommended Daily Run

```bash
bun run find:tags --min-cell 4
bun run generate:board --seed 2026-04-12 --min-cell 4 --attempts 5000 --boards 1
bun run confirm:board --board-index 0
```

## Notes

- Tags are objective (alcohol, volume, container, taste clock, etc.).
- Subjective dish-pairing taste tags are excluded.
- If too few boards are found:
  - Lower `--min-cell` (for example from 4 to 3), or
  - Increase `--attempts`.

## Release/Packaging

- These scripts are developer tooling only.
- Do not import anything from `scripts/` in app runtime code.
- Do not use these scripts in the mobile/web production bundle.
- Keep board generation in build/dev/admin workflows, then ship only generated data needed by the app.

## Step 4: Seed to Firestore

Pushes the top board from `data/generated-boards.json` to `boards/{YYYY-MM-DD}` in Firestore.

```bash
# Provide service account credentials one of two ways:
# 1. JSON contents in env (preferred for CI):
export FIREBASE_SERVICE_ACCOUNT_KEY='{"type":"service_account",...}'
# 2. Path to a JSON file (firebase-admin reads GOOGLE_APPLICATION_CREDENTIALS):
export GOOGLE_APPLICATION_CREDENTIALS=/path/to/key.json

bun run seed:firestore -- --date 2026-05-06
bun run seed:firestore -- --date 2026-05-06 --days 30
```

Defaults: `--date` falls back to tomorrow UTC, `--days 1`, `--boards-file data/generated-boards.json` is the input.

When `--days N` is set, the script writes one document per consecutive date starting from `--date`, cycling through the available board pool round-robin. If the pool is smaller than `N`, you'll see a warning and dates will reuse boards. Generate a larger pool first with `bun run generate:board --boards 30` (and a fresh `products.json`) for unique daily puzzles.

Output:
- Firestore documents `boards/{YYYY-MM-DD}` with `rows`, `cols`, `score`, `counts`, `seed`, `generatedAt`.

## Cron via GitHub Actions

A workflow lives at `.github/workflows/seed-board.yml`. It runs nightly at 22:00 UTC (~midnight Europe/Stockholm) and writes tomorrow's board to Firestore. It can also be triggered manually from the Actions tab.

Each run:
1. Fetches a fresh `products.json` from the community catalog mirror at `https://susbolaget.emrik.org/v1/products` (CORS-open, no auth, ~100MB).
2. Runs Step 1 (`find:tags`) and Step 2 (`generate:board`) in CI.
3. Runs Step 4 (`seed:firestore`) to write `boards/{YYYY-MM-DD}` in Firestore.

The workflow is configured for github.com Actions runners — KTH GHE may not have Actions enabled, so push to a github.com mirror if needed.

Repository secret required:
- `FIREBASE_SERVICE_ACCOUNT_KEY` — raw JSON of a Firebase service account. Scope it to Firestore writes on the `boards` collection if your IAM allows.
